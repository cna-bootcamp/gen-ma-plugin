  1. "3계층 활성화 구조"는 실제와 다름 (113-138행)

  현재 표준은 런타임 상주파일 → Core 스킬 → 특정 스킬을 3계층이라 설명하지만,
  실제 활성화 경로는 여러 가지이며 Core를 거치지 않는 경우가 대부분임:
  ┌─────────────────────────────────────┬───────────────────────────┬───────────┐
  │                경로                 │           흐름            │ Core 경유 │
  ├─────────────────────────────────────┼───────────────────────────┼───────────┤
  │ 슬래시 명령                         │ 런타임 → 특정 스킬        │     X     │
  ├─────────────────────────────────────┼───────────────────────────┼───────────┤
  │ 자연어 (스킬 자체 description 매칭) │ 런타임 → 특정 스킬        │     X     │
  ├─────────────────────────────────────┼───────────────────────────┼───────────┤
  │ 모호한 요청 (복잡한 의도 판별)      │ 런타임 → Core → 특정 스킬 │     O     │
  └─────────────────────────────────────┴───────────────────────────┴───────────┘
  2. 라우팅 테이블은 기술적 필수가 아님 (119-123행)

  - 슬래시 명령: 런타임이 skills/ 디렉토리 스캔으로 자동 발견
  - 자연어 라우팅: 각 스킬의 frontmatter description + ## 활성화 조건으로 LLM이 자체 매칭 가능
  - 순환 의존 논거: 런타임이 스킬 메타데이터를 시작 시 로드하면 순환 없음

  3. CLAUDE.md 등록은 토큰 낭비 (457-491행)

  - 플러그인을 사용하지 않는 세션에서도 비용 발생
  - 대안: help 유틸리티 스킬로 대체하면 호출 시에만 토큰 사용
